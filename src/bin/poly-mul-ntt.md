# Number Theoretic Transform (NTT)

## Overview

This session closed the loop opened by two earlier ones. The naive polynomial multiplication session established that multiplying two polynomials by direct convolution costs O(n·m), and explicitly flagged cyclic versus linear convolution as a distinction NTT would later need. The primitive roots session built, as its final act, the ability to construct an element of any desired order dividing p-1 modulo a prime p — and named NTT as the reason that construction mattered. This session used both: the strategy of evaluating polynomials at cleverly chosen points instead of multiplying coefficients directly, and the specific points (roots of unity mod a prime) that make that strategy fast rather than merely correct.

## Constructing a primitive n-th root of unity mod p

Given a prime p and a primitive root g of p (so g has order exactly p-1), and given a divisor n of p-1, the session first re-derived a fact that turns out to be foundational to everything that follows: the element

$$h = g^{(p-1)/n} \pmod p$$

has order exactly n. The proof leans on a general lemma proved in the same step: since g has order p-1, g^k ≡ 1 (mod p) if and only if (p-1) divides k. Applying this with k = ((p-1)/n)·e shows h^e ≡ 1 (mod p) exactly when n divides e, so the smallest positive e achieving this is e = n itself — meaning h's order is exactly n, not some smaller divisor of n. This h plays the role in NTT that e^(2πi/n) plays in the ordinary complex FFT: a "primitive n-th root of unity," but living inside the finite field Z/pZ instead of the complex numbers.

## Two representations of a polynomial, and why they multiply differently

A polynomial of degree less than n can be represented two ways: as a list of n coefficients, or as its values at any n distinct points. The session established, using the same "a nonzero degree-k polynomial has at most k roots over a field" fact used earlier to prove primitive roots exist, that a polynomial of degree at most d is uniquely determined by its values at any d+1 distinct points — no two different polynomials of that degree can agree at that many points, since their difference would have too many roots for its degree.

This matters for multiplication because of a second, more elementary fact: if C(x) = A(x)·B(x) as polynomials, then for any specific number x0, C(x0) = A(x0)·B(x0) — substituting a number into a product of polynomials is just multiplying the two substituted values. Combining both facts gives a strategy: to compute C = A·B, pick n ≥ deg(A) + deg(B) + 1 arbitrary distinct points, evaluate A and B at each one, multiply the values pairwise (n cheap scalar multiplications), and interpolate the n resulting points back into a polynomial. Because the true C already agrees with those n points, and only one polynomial of the right degree can agree with all of them, interpolation is guaranteed to recover the true C exactly.

Tried with arbitrary points, though, this strategy is not obviously worthwhile: evaluating a degree-(n-1) polynomial at one point costs O(n) via direct substitution, so evaluating at n points costs O(n²) — no better than direct convolution — and interpolation through n arbitrary points costs O(n²) or worse by Gaussian elimination on the resulting linear system. The payoff only appears once the points are chosen so evaluation itself can be done faster than one-at-a-time.

## Fast evaluation: splitting by parity and the pairing property

Write P(x) = E(x²) + x·O(x²), where E collects P's even-indexed coefficients and O its odd-indexed coefficients. Evaluating P at a point x and at -x shares nearly all of its work, since (-x)² = x² means E and O only need to be evaluated once, at x², to produce both P(x) = E(x²) + x·O(x²) and P(-x) = E(x²) - x·O(x²). With ordinary real numbers this trick only pays off once: squaring two real numbers that were negatives of each other produces the *same* value, not a new pair of negatives, so the recursion cannot continue.

The primitive n-th root of unity h fixes this. Because h has order exactly n, h^(n/2) squares to 1 (since (h^(n/2))² = h^n = 1) and, by the "x² ≡ 1 mod p has only the two solutions ±1" fact from the Miller-Rabin session, h^(n/2) must be -1 (it cannot be 1, since n/2 < n and n is the *smallest* exponent giving 1). This means the n points h^0, h^1, ..., h^(n-1) pair up as h^k and h^(k+n/2) = -h^k — genuine ± pairs, mod p. Squaring collapses these n points onto the n/2 points that are themselves powers of h², which has order n/2 (a direct application of the same order-of-a-power lemma used to build h in the first place, applied recursively). Since h² is again a primitive root of unity — just of half the order — the same pairing trick applies one level down, and recurses all the way to a single point.

This recursion bottoms out cleanly only if n is a power of two. The session verified this concretely with p=7, h=3 (order 6, not a power of 2): the first level of pairing (n/2 = 3) works, since h^3 ≡ -1 mod 7, but the resulting 3-element sub-problem cannot be halved again, because a group of order 3 has no element of order 2 — by Lagrange's theorem, an order-2 element would require 2 to divide 3, which it does not. Squaring the 3 remaining points merely permutes them among themselves rather than collapsing them further. So n must be chosen as a power of 2 dividing p-1, and p must be a prime whose p-1 has enough factors of 2 to support the desired n.

## The inverse transform

Given the n values of a polynomial at h^0, ..., h^(n-1), the coefficients can be recovered by running the same recursive procedure used for the forward transform, but with h^(-1) in place of h, and then dividing every entry of the result by n modulo p. Establishing why this works rests on one additional algebraic fact, proved here and used in the Correctness section below: for any integer m,

$$\sum_{j=0}^{n-1} h^{jm} \equiv \begin{cases} n \pmod p & \text{if } n \text{ divides } m \\ 0 \pmod p & \text{otherwise} \end{cases}$$

This identity says that summing the n-th roots of unity, each raised to a common power m, collapses to something very simple: either every term equals 1 (when n divides m, so h^m itself is 1) and the sum is just n copies of 1, or the terms are spread evenly around the roots of unity and cancel completely. The proof is a "multiply the sum by a ratio and telescope" trick: letting T denote the sum, h^m·T - T telescopes to h^(mn) - h^0 = (h^n)^m - 1 = 0, so T·(h^m - 1) ≡ 0 (mod p). Since p is prime, this forces either T ≡ 0 or h^m ≡ 1, and h^m ≡ 1 exactly when n divides m — the same order lemma used throughout this session.

## Implementation notes and bugs found

The recursion was implemented in place, addressing the shared coefficient array via an (offset, stride) pair rather than allocating fresh sub-arrays: the even-indexed branch recurses with the same offset and double the stride, the odd-indexed branch with offset+stride and double the stride. Several distinct bugs surfaced while getting this right, each worth remembering. First, the base case was checked as `size == 1` instead of `size == 0`, causing the recursion to stop one level too early and skip a real combine step. Second, once fixed, the combine step's read positions were computed assuming E's and O's outputs occupied contiguous halves of the local array (as they would with fresh sub-arrays), when in fact — because of the in-place offset/stride scheme — they are interleaved, living at offset + i·stride·2 and offset + stride + i·stride·2 respectively. Third, fixing the read positions exposed a genuine data hazard: with size ≥ 2, one iteration's write to a "high" position lands exactly on the position a later iteration still needs to read, corrupting it before it's used; no loop order avoids this in general (checked concretely both forward and reverse), so the fix was to decouple reads from writes entirely — computing all of a call's outputs into a small scratch buffer before copying anything back into the shared array. Fourth, the twiddle factor root^k was initially computed as a single fixed value per call rather than incrementing with the loop index k; the eventual fix precomputed a table of h^0, ..., h^(n-1) once and walked an index into it by +stride per iteration (since root_at_depth = h^stride, advancing the table index by stride advances the represented exponent by stride, producing root_at_depth^k incrementally). Fifth, at the convolution level, padding a polynomial to a longer transform length must add *trailing* zero coefficients (extra high-degree terms, which don't change the polynomial), not *leading* zeros, which would silently multiply the polynomial by a power of x.

## Correctness

The recursive evaluation procedure — call it ntt(coefficients, root) — splits its input by parity into E and O, recurses on each with root² (evaluating each at the n/2 points (root²)^0, ..., (root²)^(n/2-1)), and combines via

$$\text{result}[k] = E_{\text{result}}[k] + \text{root}^k \cdot O_{\text{result}}[k], \qquad \text{result}[k+n/2] = E_{\text{result}}[k] - \text{root}^k \cdot O_{\text{result}}[k]$$

Its correctness follows by induction on the recursion. The base case (a single coefficient) is a degree-0 constant, which evaluates to itself at any point, so returning it unchanged is correct regardless of what root is. For the inductive step, assume (by induction) that E_result[k] = E((root²)^k) and O_result[k] = O((root²)^k) for k = 0, ..., n/2-1. Substituting x = root^k into P(x) = E(x²) + x·O(x²) gives P(root^k) = E(root^(2k)) + root^k·O(root^(2k)) = E_result[k] + root^k·O_result[k], which is exactly result[k] as defined by the combine step. The paired output follows the same way using root^(k+n/2) = -root^k: P(-root^k) = E(root^(2k)) - root^k·O(root^(2k)) = result[k+n/2]. So the combine step produces exactly the true values of P at every one of its n points, for every level of the recursion.

This same argument establishes the inverse transform's correctness too, since running the identical procedure with h^(-1) in place of h is just the forward transform applied to a different (but equally valid) primitive n-th root of unity — h^(-1) also has order exactly n, by the same order-of-a-power reasoning used to construct h. Substituting the forward transform's definition into the inverse formula produces a double sum that collapses, via the orthogonality identity from the previous section, to exactly n times the original coefficient — dividing by n at the end recovers it exactly.

## Complexity

The recursion satisfies T(n) = 2T(n/2) + O(n): two half-size recursive calls, plus O(n) total combine work across all pairs at that level. Solving by counting level by level — there are log₂(n) levels, and each does O(n) total work — gives T(n) = O(n log n). This is a different outcome from Karatsuba's T(n) = 3T(n/2) + O(n): Karatsuba's branching factor (3) exceeds its halving factor (2), landing in the Master theorem's "recursive work dominates" case and giving O(n^log₂3) ≈ O(n^1.585); NTT's branching factor (2) exactly equals its halving factor, the Master theorem's boundary case, giving the extra log(n) factor instead of a higher power. Multiplying two polynomials via NTT costs three transforms (two forward, one inverse) plus O(n) pointwise multiplication, all O(n log n) — a substantial asymptotic improvement over poly-mul-naive's O(n²).

## Edge cases

The transform computes the convolution exactly modulo p, but recovering the true (un-reduced) integer coefficients of a product requires the chosen modulus p to exceed every true coefficient of that product. If the true convolution value at some position exceeds p, it wraps around modulo p and disagrees with the exact integer answer, even though the NTT computation itself is completely correct modulo p at every step. This is not a flaw in the transform; it is a modulus-sizing requirement that must be checked by whoever picks p for a given multiplication, analogous to allocating enough chunks in schoolbook or Karatsuba multiplication to avoid truncating the true product.

## Worked example

Take p=5 and h=2, which has order 4 (2, 4, 3, 1 before returning to 1), so n=4 is usable directly (h = g here, since n = p-1). Forward-transform P(x) = 1 + 2x + 3x² + x³: split into E=[1,3] and O=[2,1]. Recursing on E with root²=4 gives base cases [1] and [3], combining (only k=0 exists at this size) to [1+4^0·3, 1-4^0·3] = [4,3]. Recursing on O with root²=4 similarly gives [2+1, 2-1] = [3,1]. Back at the top with root=2: k=0 gives [4+2^0·3, 4-2^0·3] = [7,1] ≡ [2,1] mod 5, and k=1 gives [3+2^1·1, 3-2^1·1] = [5,1] ≡ [0,1] mod 5. Interleaving correctly (result[0], result[1], result[2], result[3]) gives values [2, 0, 1, 1] — matching direct evaluation of P at 1, 2, 4, 3 respectively.

To invert: h^(-1) = 3 (since 2·3 ≡ 1 mod 5) and n^(-1) = 4 (since 4·4 ≡ 1 mod 5). Running the identical procedure on [2,0,1,1] using root=3 in place of 2 gives [4,3,2,4] (following the same split-recurse-combine steps), and multiplying every entry by n^(-1)=4 gives [16,12,8,16] mod 5 = [1,2,3,1] — recovering P's original coefficients exactly.
