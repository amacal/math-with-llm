# Trial Division Factorization

## Overview

This session computes the complete prime factorization of a positive integer n from scratch, without leaning on the Pollard's-Rho machinery already built in `mod-pollard-rho.rs` and `mod-primitive-root.rs`. The core idea in one sentence is that at every stage we take whatever value currently remains, pull out its smallest divisor greater than one, divide it out, and repeat until nothing but one is left. What makes this session distinctive is that the algorithm is not new; it is the existence half of the Fundamental Theorem of Arithmetic proof, from Section 2.1 of *Number Theory Step by Step*, read directly as a procedure — that proof showed a composite number always has a smallest divisor greater than one, and that this smallest divisor is always prime, and turning that single fact into a loop is the entire algorithm. This session runs a little past the usual length band because it independently re-derives several supporting facts from scratch (the square-root search bound, the safety of never backtracking the search index, and the two-invariant correctness argument) rather than citing them, and closes with a genuine empirical result from testing the implementation on values near u64::MAX.

## The smallest divisor and why it is prime

Fix an integer n greater than one and consider the set of its nontrivial divisors,
$$N = \{d : d \text{ divides } n,\ d \neq 1,\ d \neq n\}.$$
This set is empty exactly when n is prime, and nonempty exactly when n is composite, since being composite forces some other divisor to exist. Because N is a set of positive integers, the Well-Ordering Principle — the same principle behind the existence proof in Section 2.1 — guarantees that whenever N is nonempty it has a least element; call it d.

This least element d must itself be prime, by a contradiction on minimality. Suppose d were composite, with its own divisor x satisfying
$$1 < x < d.$$
Because x divides d and d divides n, x also divides n, by the transitivity of divisibility used in the earlier gcd sessions in this repo. Since x is strictly between one and d, and d is strictly less than n, x qualifies as a member of N — but x is smaller than d, contradicting that d was the *smallest* member of N. This is only avoidable if d was never composite, so d must be prime.

## Extending one factor into a full factorization

A single smallest divisor only peels off one prime. To get the complete factorization, the same fact is applied again to the quotient: once d is removed and the remaining value becomes n divided by d, that value's own smallest divisor is again prime by the identical argument, and dividing it out again shrinks the value further. Repeating this until the remaining value reaches one produces the full list of prime factors, with multiplicity, by recording every d extracted along the way.

One refinement found this session is that the search for the *next* divisor does not need to restart at two; it can resume from the same d that just succeeded. Write m for the remaining value and suppose some a less than d divides the new quotient m divided by d. Since m equals d times this quotient, a dividing the quotient means a also divides m, again by transitivity of divisibility — so a would be a divisor of m smaller than d, contradicting that d was m's smallest divisor greater than one. No divisor of the quotient can ever be smaller than d, so the search index only ever moves forward across the whole run.

## The square-root bound

Trial division as described so far would, in the worst case, search every candidate up to the remaining value itself. A sharper bound follows from the same style of minimality argument: suppose m is composite with smallest divisor d greater than one, and suppose for contradiction that d exceeds the square root of m. Writing m as d times its complementary factor, dividing both sides of d squared being greater than m by d gives
$$\frac{m}{d} < \sqrt{m} < d.$$
So m divided by d is itself a divisor of m, strictly smaller than d — contradicting that d was the smallest divisor of m. Hence the smallest divisor of any composite value never exceeds its square root, and if trial division has tested every candidate up to the square root without success, the remaining value must be prime. This is the same conclusion the book reaches in Section 2.2 via Proposition 2.9, derived here independently for the specific smallest-divisor case this algorithm needs.

## Correctness

The loop maintains two invariants at every iteration. First, divisor is always less than or equal to the smallest prime factor of whatever value currently remains, since the search index only advances once a candidate is exhausted, never skipping past a valid one. Second, the product of every value already collected into the output, times whatever remains, always equals the original input n, since each division step replaces one factor in this product (the remaining value) by two factors of identical product (the extracted d and the new, smaller remaining value).

These two invariants combine with a strict-descent termination argument, identical in shape to the one used in Section 2.1: the remaining value strictly decreases with every division, and a strictly decreasing sequence of positive integers cannot continue forever, so the loop must eventually stop, in one of two ways. If the remaining value reaches exactly one, the product invariant immediately says the collected output already multiplies back to n. If instead the loop exits early because the search index's square exceeds the remaining value, the square-root bound from the previous section says the remaining value's smallest factor, if it were composite, would have to be less than or equal to its square root, which is strictly less than divisor by the exit condition — while the first invariant says divisor is less than or equal to that same smallest factor. Chaining these gives a value strictly less than itself, which is impossible, so the remaining value must be prime. Appending it as the final factor preserves the product invariant and completes a correct factorization either way.

## Complexity

A single stage of the algorithm, in the worst case that the remaining value turns out to be prime, costs on the order of its square root in trial divisions, since the search only stops once the index's square exceeds it. Because the search index never resets and the remaining value shrinks by at least a factor of two with every successful division, the total cost across the entire factorization is also bounded by O(sqrt(n)), dominated by whichever single stage corresponds to the largest prime factor present. This is a real practical cost: for n near u64::MAX, roughly 2^64, the square root is roughly 2^32, versus the Pollard's-Rho approach already in `mod-pollard-rho.rs` and `mod-primitive-root.rs`, whose O(n^(1/4)) cost is only about 2^16 for the same magnitude.

Running this factorizer over the one hundred integers immediately below u64::MAX confirmed the bound is not just theoretical: three of those integers turned out to be prime outright, forcing the full roughly 2^32 search, and one other had 2^61 minus 1, a Mersenne prime, as its largest factor, landing in the same worst-case regime. By contrast, one integer in the same range factored into a long list of small primes and finished essentially instantly — confirming that the O(sqrt(n)) bound is driven by the size of the largest prime factor present, not by the raw magnitude of n.

## Edge cases

The value n equal to zero has no meaningful factorization, since zero equals zero times any integer whatsoever, so the function returns None rather than an empty or arbitrary vector. The value n equal to one is different: under the convention that an empty product of primes equals one, it has a well-defined answer, an empty list of factors, so it returns an empty vector wrapped in Some, keeping "no factorization exists" and "the factorization happens to be empty" visibly distinct. The remaining edge case is overflow: since the search index and the remaining value are both u64, checking whether the index squared exceeds the remaining value cannot be computed directly once the index grows past roughly 2^32, since squaring it would overflow. The implementation uses a checked multiplication and treats overflow exactly like an ordinary "exceeds the remaining value" result, sound because the remaining value is bounded above by u64::MAX, so an overflowing square is automatically larger than anything it could hold — the identical pattern used earlier in `mod-isqrt-bisect.rs`'s bisection search.

## Worked example

Consider factoring n equal to twelve by hand, tracing the same steps carried out during the session. The loop begins with the search index at two and the remaining value at twelve; since two divides twelve, two is pushed onto the output and the remaining value drops to six, after which the loop checks whether the search index squared, four, exceeds the new remaining value six, and since it does not, the loop continues without advancing the search index. On the next pass the search index is still two, and since two also divides six, two is pushed onto the output a second time and the remaining value drops to three; now the search index squared, four, does exceed the remaining value three, so the loop breaks. At this point the output holds two and two, and the remaining value three is greater than one, so three is appended as the final leftover prime, giving the complete factorization two, two, three. Multiplying these back together, two times two times three equals twelve, confirming the product invariant held throughout. This trace also illustrates the resume-without-backtracking behavior: the search index stayed at two across two successful divisions before ever needing to advance, which is exactly what lets the algorithm catch a repeated factor without any special-casing for perfect squares or higher powers.
