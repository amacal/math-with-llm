# Primitive Roots mod p

## Overview

This session asks a question sitting quietly underneath Baby-Step Giant-Step: given a prime p, is there always some element whose powers sweep out the entire multiplicative group before repeating, and if so, how do you find one efficiently? The group (Z/pZ)* consists of the residues 1 through p-1 under multiplication mod p, and always has exactly p-1 elements, since every nonzero residue mod a prime is coprime to p. The question of primitive roots is the question of whether this group has a *generator* — a single element from which every other element can be produced by repeated multiplication with itself.

The right way to see this concretely is to watch two elements behave under repeated multiplication mod the same prime. Take p = 7, so the group is {1, 2, 3, 4, 5, 6}. Multiplying 3 by itself repeatedly gives 3, 2, 6, 4, 5, 1 — all six elements appear before the sequence returns to 1. Multiplying 2 by itself instead gives 2, 4, 1, 2, 4, 1, ... — only three distinct values ever appear. The element 3 is a primitive root of 7; the element 2 is not. What distinguishes them is the length of the cycle each traces before returning to 1.

## Order, and what a primitive root actually is

The **order** of an element g modulo p is the smallest positive integer k such that g^k is congruent to 1 mod p. For g = 3 and p = 7 the order is 6; for g = 2 the order is 3. A **primitive root** mod p is an element whose order equals p-1, the full size of the group — matching the two examples exactly. Before this order can be pinned down efficiently, a structural fact is needed: Lagrange's theorem, encountered earlier when proving Euler's theorem, states that the order of any element of a finite group must divide the size of the group. Applied here, the order of any g in (Z/pZ)* must divide p-1, which turns "compute the order of g" from an open-ended search into a search over the small set of divisors of p-1.

## Characterizing a primitive root without brute force

Directly computing the order of g by trying every exponent up to p-1 costs O(p) multiplications per candidate — too slow for the 64-bit primes this file targets. Lagrange's theorem fixes this a second way: since the order of g must divide p-1, the order fails to be p-1 exactly when it divides some proper divisor instead, and every proper divisor of p-1 divides p-1 over at least one of its prime factors. Writing p-1 = q1^a1 * q2^a2 * ... * qk^ak for distinct primes qi, any proper divisor is missing at least one full power of some qi, so it divides (p-1)/qi for that i. It is therefore enough to check the handful of exponents (p-1)/qi, one per distinct prime factor of p-1, rather than every divisor.

$$g \text{ is a primitive root mod } p \iff g^{(p-1)/q} \not\equiv 1 \pmod{p} \text{ for every prime } q \text{ dividing } p-1$$

The direction of this check is easy to misstate, and it was misstated once during this session before being corrected: g is rejected when some g^((p-1)/q) *equals* 1, not when it fails to. Equality at any one of these exponents means the order of g divides (p-1)/q, strictly smaller than p-1. Only when none of the checks produce 1 is the order forced to be the one divisor of p-1 that survives all of them: p-1 itself. This test specializes neatly to a case seen already: for a safe prime, where p-1 = 2q for a prime q, the two distinct prime factors of p-1 are 2 and q, so the iff characterization collapses to the two-check shortcut from the Baby-Step Giant-Step session — checking only g^2 and g^((p-1)/2) — which that session justified directly rather than as an instance of this more general theorem.

## Why a primitive root always exists

An algorithm that searches for a primitive root is only correct if one is guaranteed to exist. The argument is a counting argument built on one borrowed fact: a polynomial equation x^k ≡ 1 (mod p) has *at most* k solutions mod a prime p, since Z/pZ is then a field and a nonzero degree-k polynomial over a field has at most k roots. Suppose, for contradiction, that no element of (Z/pZ)* has order p-1. Then by Lagrange's theorem every element's order is a proper divisor of p-1, so by the previous section's argument every element satisfies x^((p-1)/qi) ≡ 1 for at least one prime factor qi of p-1. For p = 7, the prime factors of p-1 = 6 are 2 and 3, giving x^3 ≡ 1 (at most 3 solutions) and x^2 ≡ 1 (at most 2 solutions) — together accounting for at most 5 elements, yet the group has 6. At least one element satisfies neither equation, so its order cannot divide 3 or 2; by Lagrange's theorem its order still divides 6, and the only divisor of 6 that divides neither 3 nor 2 is 6 itself. That element has order exactly 6, contradicting the assumption. The same counting argument generalizes to any prime p, so an element of order exactly p-1 always exists.

## The search algorithm

The characterization and existence proof combine into a concrete algorithm. First, `factorize(p-1)` produces the distinct prime factors of p-1, reusing Miller-Rabin for primality testing and Pollard's Rho for splitting composites, with a new wrinkle: a single Pollard's Rho call only guarantees a *nontrivial* factor d of n, not a *prime* one. An early version of `factorize` pushed d directly into the result without checking primality, and on inputs like 15785 it silently returned composite entries such as 55 = 5 × 11. The fix treats factoring as recursive splitting: maintain a stack of numbers still needing resolution, and whenever Pollard's Rho returns a factor d of some n on the stack, push both d and n/d back instead of accepting d as final; only a number that passes the primality test is added to the result set (kept in a HashSet, since recursive splitting can rediscover the same prime along different branches).

Second, `primitive_root(p)` searches candidates g = 2, 3, 4, ... and returns the first that passes the characterization. A second bug surfaced here: the initial implementation never verified p itself was prime before searching. Since the entire theory — Lagrange's theorem, the existence proof, even the definition of the group being searched — depends on p being prime, running the search against a composite p produces a value that satisfies the code's check but means nothing mathematically; feeding the function consecutive odd numbers near u64::MAX exposed this, since the search kept returning g = 2 far more often than chance would suggest. The fix runs `miller_test(p)` at the top of `primitive_root` and returns `None` immediately if p is not prime.

## Correctness

`primitive_root(p)` is correct because every check it performs is backed by a proof rather than a heuristic. The rejection test is exactly the iff characterization derived above, so g is accepted only when its order is forced to be the full p-1. The search is guaranteed to terminate with a correct answer, rather than exhausting every candidate and returning nothing, because the existence argument shows some primitive root is always present among {2, ..., p-2} for a genuine prime p. Both bugs found during the session were correctness gaps in the supporting machinery rather than in the test itself — an unverified `factorize` result could hand the test a composite pretending to be a prime factor of p-1, and an unverified p could send the whole group-theoretic argument searching over a set that is not actually a group of order p-1 at all. Fixing both restores the guarantee that a returned g is a genuine primitive root, and that `None` means only that p itself was not prime.

## Complexity

The dominant cost is factoring p-1. Pollard's Rho splits a composite n in expected O(n^(1/4)) time, and `factorize` may need up to log2(p-1) splits in the worst case, since the smallest possible prime factor is 2. Each split is checked with `miller_test` at O(log n) per check, so `factorize(p-1)` costs O((p-1)^(1/4) log p). The candidate search contributes a smaller second term: each candidate costs O(log^2 p) (one `mod_exp` call per distinct prime factor of p-1, of which there are at most log2(p-1)), and the expected number of candidates tried is the reciprocal of the density φ(p-1)/(p-1). Writing (p-1)/φ(p-1) as a product of q/(q-1) over the distinct prime factors q of p-1, its logarithm behaves like the sum of 1/q over those primes, which by the same Mertens'-theorem reasoning used in the Sieve of Eratosthenes session grows no faster than O(log log p). So the candidate search costs O(log log p · log^2 p) in expectation.

$$T(p) = O\!\left((p-1)^{1/4} \log p\right) + O(\log\log p \cdot \log^2 p) = O\!\left(p^{1/4} \log p\right)$$

The first term dominates for every p of practical size, since p^(1/4) eventually outgrows any power of log p. The overall expected cost of `primitive_root(p)` is therefore O(p^(1/4) log p), inherited almost entirely from factoring p-1.

## Edge cases

The search starts at g = 2 rather than g = 1 or scanning downward from p-1, and both choices are deliberate. g = 1 always has order 1, since 1^1 ≡ 1 trivially, so it can never be a primitive root except in the degenerate case p = 2 — a case this implementation excludes via an explicit guard on p ≤ 2, rather than silently returning a wrong answer. Scanning downward fares no better: g = p-1 is congruent to -1 mod p, and (-1)^2 ≡ 1 always, so p-1 always has order exactly 2 (except when p = 3, where p-1 = 2 coincides with the group order). Neither starting point has a *general* advantage beyond avoiding these two guaranteed failures, because the density of primitive roots among all residues — the fraction φ(p-1)/(p-1) — is a property of the whole set, not of where in the range one starts looking.

## Worked example

Take p = 7, so p-1 = 6 and factorize(6) = [2, 3]. The search starts at g = 2. Testing the two exponents: (p-1)/2 = 3, and 2^3 mod 7 = 1, so the check fails immediately — 2 is rejected, consistent with the order-3 cycle observed by hand at the start of the session. Moving to g = 3: (p-1)/2 = 3 gives 3^3 mod 7 = 6, not 1, and (p-1)/3 = 2 gives 3^2 mod 7 = 2, also not 1. Both checks pass, so 3 is returned as the primitive root of 7 — matching the direct computation of powers of 3 mod 7 (3, 2, 6, 4, 5, 1) cycling through all six nonzero residues before returning to 1.
