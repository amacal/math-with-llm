You're auditing the prompts, not the user, not any single session's math content. Don't talk to the user. Don't edit any file. Return findings as text only.

## Steps

1. Read the last N history entries in full (not just INDEX.yml) and check three things per session: (a) whether `corrections`, `bugs_found`, or `not_completed` needed an unusually long list, a calibration signal; (b) whether INDEX.yml's `prerequisite_index` for this session, cross-checked against its `corrections`/`bugs_found`, suggests a mid-session backtrack to an earlier concept (e.g. a correction restating a fact a genuine prerequisite should already have supplied) — a sign the plan agent under-flagged a missing prerequisite; (c) whether `open_questions`/`not_completed` repeatedly names the same kind of gap across sessions, pointing at friction the current field set has no dedicated place to record. Don't read hedging language in the notes file's prose as a proxy for this — notes are pure math content and shouldn't carry session-friction signals at all; that evidence belongs in HISTORY's fixed fields, and its absence there is itself a finding.

2. For each rule in CLAUDE.md's "Teaching style" and "Pacing and assumed knowledge", look for concrete evidence in the last N sessions that the rule was followed AND that following it produced a session closing with a genuine correctness/complexity argument (not just a completed one). No discoverable evidence of impact in N sessions → candidate for rewording or removal.

3. Check whether session-select's proposals over the last N sessions actually spanned domains as intended (`.skills/session-select.md`'s "Force genuine breadth" step), or whether breadth was suggested but the user always picked chain-continuation — a signal the rule needs to be stronger (e.g. requiring justification when breadth is declined), not just present.

4. Check whether any notes file needed to violate the 700–1500 word target with a stated justification more than once for the same reason (repeatedly "thin wrapper," repeatedly "large synthesis") — a recurring pattern means the target band itself may need adjusting, not continued per-session exceptions.

5. Identify up to 3 specific, minimal wording changes to CLAUDE.md or a `.skills/*.md` file addressing a gap found in steps 1-4, each justified by a specific citation to a session where the current wording's absence caused friction, ambiguity, or a missed opportunity. More than 3 evidence-backed issues found → rank by friction/ambiguity caused, report only the top 3; batch remaining valid-but-lower-priority findings into "unverified ideas" alongside genuinely speculative suggestions, clearly distinguishing the two kinds. Never propose a change unsupported by cited history evidence — speculative improvements with no session evidence never count toward the 3.

6. Report back: (a) up to 3 evidence-backed wording changes with citations, (b) any unverified/speculative ideas worth trying but not yet evidenced, (c) a one-line verdict on whether the current prompt set is keeping pace with the user's growth, or is now too slack (redundant reps) or too tight (recurring friction) for the current stage.
